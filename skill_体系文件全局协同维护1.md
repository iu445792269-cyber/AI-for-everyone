# Skill: 体系文件全局协同维护

## Metadata
- **Name:** 体系文件全局协同维护
- **Version:** 2.0
- **Type:** workflow-skill
- **Owner:** User
- **Scope:** 企业质量/管理/作业/流程/制度类体系文件的跨文件检索、定位、逐条确认、批量修订与留痕

---

## 1. 背景与目标
在日常维护体系文件时，同一句条款或同一类表述常同时出现在多份文件中。人工逐个检索、定位、核对并修改，容易出现以下问题：
- 漏改：没有找全所有相关文件；
- 错改：只看关键词就替换，忽略上下文语义；
- 失控：批量改写后无法逐条追踪；
- 难审计：后续无法说明改了哪些文件、改了什么、由谁确认。

本 Skill 的目标是：
1. 用户输入“原句/原条款”和“新句/新条款”后，自动在指定文件范围内检索所有相关文件；
2. 将命中的文件逐一列出，并明确标出该语句在文件中的位置、上下文和命中方式；
3. 在每一条实际写入前，必须先向用户确认；
4. 用户确认后再执行修改，并生成标准化修订日志；
5. 出现风险时支持跳过、手动改写、回滚和终止。

---

## 2. 适用场景
适用于以下场景：
- 质量体系文件、程序文件、作业指导书、制度文件、表单模板的统一条款维护；
- 某一句管理要求、职责描述、检验标准、记录要求在多文件中的同步替换；
- 审核前、换版前、制度修订前的全库排查与批量更新；
- 需要“检索 → 定位 → 对比 → 人工确认 → 批量写入 → 留痕”的高可靠修订流程。

不适用于以下场景：
- 仅凭模糊意图直接整库自动改写；
- 无法读取或无法安全回写的加密/受限文件；
- PDF 等天然不适合原位编辑的文件类型（默认仅检索和定位，不直接覆写）。

---

## 3. 核心原则

### 3.1 先检索，后修改
任何写入动作之前，必须先完成全局检索、定位与候选清单生成。

### 3.2 先展示，后确认
每一条命中项都必须先展示“文件名 + 位置 + 上下文 + 修改前后对比”，再等待用户确认。

### 3.3 一条一确认
默认逐条确认，不允许未经确认直接全量覆写。

### 3.4 仅改命中项，不做扩大解释
若某段内容与目标句高度相似但并非同一条款，不自动写入，只能作为“疑似项”交由用户确认。

### 3.5 先备份，再写入
任何文件在修改前必须创建备份副本，确保可回滚。

### 3.6 全程留痕
检索结果、确认结果、修改结果、跳过项、失败项、回滚记录都必须写入修订日志。

---

## 4. 输入定义

### 必填输入
- `Target_Text`：待检索并准备替换的原句/原条款
- `Replacement_Text`：拟替换成的新句/新条款

### 选填输入
- `Search_Scope`：检索范围，如指定目录、知识库、标签、部门文件夹
- `File_Types`：允许处理的文件类型，如 `.docx`、`.xlsx`、`.txt`、`.md`
- `Match_Mode`：匹配模式，可选：`exact`、`fuzzy`、`semantic`、`hybrid`
- `Confirm_Mode`：确认模式，可选：`one_by_one`、`file_by_file`
- `Risk_Level`：风险等级，可选：`strict`、`balanced`
- `Create_Backup`：是否备份，默认 `true`
- `Dry_Run`：是否只预览不写入，默认 `false`

---

## 5. 输出定义

### 5.1 检索阶段输出
输出《命中清单》，至少包含：
- 文件名
- 文件路径
- 文件类型
- 命中位置（章节、页码、段落、单元格、行列号等）
- 命中内容
- 前后文摘要
- 匹配方式（精确匹配 / 模糊匹配 / 语义匹配）
- 风险提示（如“疑似相似句”“可能多处命中”）

### 5.2 审阅阶段输出
对每一条命中项输出《修改预览卡》，至少包含：
- 当前文件
- 具体位置
- 修改前
- 修改后
- diff 对比
- 操作选项

### 5.3 执行阶段输出
输出《执行结果》，至少包含：
- 已修改条数
- 已跳过条数
- 手动改写条数
- 修改失败条数
- 每个文件的处理结果

### 5.4 归档阶段输出
输出《体系文件批量修订报告》，至少包含：
- 任务编号
- 执行时间
- 操作人
- 检索范围
- 原条款与新条款
- 命中文件总数
- 实际修改清单
- 跳过清单
- 失败清单
- 备份位置
- 回滚信息（如发生）

---

## 6. 工作流

### Step 1：接收任务
Agent 接收用户输入的原条款与新条款，并进行基础校验：
- 是否为空；
- 原条款与新条款是否完全相同；
- 是否超出允许处理的文件范围；
- 是否开启了备份与审计。

若 `Target_Text == Replacement_Text`，则直接提示用户无需修改并结束流程。

### Step 2：全局检索
在指定范围内执行多策略检索：
1. 精确匹配：用于找出完全一致的句子；
2. 模糊匹配：用于找出格式稍有变化但内容相同的句子；
3. 语义匹配：用于找出疑似同义改写条款；
4. 结构定位：提取章节号、段落号、页码、单元格坐标等位置信息。

输出第一版命中总表。

### Step 3：去重与风险分级
对检索结果进行整理：
- 相同文件同一位置重复命中时自动去重；
- 同一文件多处命中时拆成多条；
- 将“精确命中”与“疑似相似项”分开标识；
- 对风险较高的项添加提示，例如：
  - 句式相似但关键数值不同；
  - 命中内容处于表头/页眉/页脚/目录；
  - 命中内容属于历史修订记录而非正文；
  - 文件可能为模板文件，修改影响范围大。

### Step 4：生成修改预览
对每一条命中项生成修改预览：
- 原文片段；
- 新文片段；
- Before / After 对比；
- 高亮差异；
- 文件与位置；
- 必要上下文。

### Step 5：逐条确认
按顺序逐条向用户展示，每条仅允许在收到用户确认后继续。支持以下操作：
- `1`：确认修改当前条
- `2`：跳过当前条
- `3`：手动编辑当前条的新内容
- `4`：查看更长上下文
- `5`：终止本次任务

说明：
- 默认一次只处理一条；
- 未明确确认前，不写入文件；
- 若用户中途终止，则保留已完成记录并结束。

### Step 6：安全写入
用户确认后再执行写入：
- 先备份原文件；
- 再对当前命中位置进行定点替换；
- 尽量按“位置 + 内容”双重校验写入，避免同文件内误替换其他相同句子；
- 写入后重新读取并校验替换结果。

如校验失败：
- 标记为失败；
- 不继续默认扩散写入；
- 提醒用户查看该条。

### Step 7：持续迭代
自动推进到下一条命中项，重复“展示 → 确认 → 写入/跳过 → 记录”的过程，直到清单处理完毕。

### Step 8：生成报告
任务结束后自动生成《体系文件批量修订报告》，并附：
- 处理统计；
- 文件清单；
- 每条确认记录；
- 修改前后摘要；
- 失败/跳过说明；
- 备份文件索引。

## 7. 交互规范

### 7.1 标准提示模板

#### 检索完成提示
```text
检索完成：共发现 8 条命中，分布在 5 份文件中。
其中精确命中 6 条，疑似相似项 2 条。
现在开始逐条审阅。
```

#### 单条审阅提示
```text
当前文件：XXX作业指导书.docx
位置：4.2 设备点检要求，第 3 段
匹配方式：精确匹配

修改前：在设备启动前，应由班组长完成点检确认并记录。
修改后：在设备启动前，应由责任岗位完成点检确认并记录。

请选择：
1 确认修改
2 跳过此条
3 手动编辑
4 查看更长上下文
5 终止任务
```

#### 单条执行反馈
```text
当前条处理完成：已修改
备份文件：XXX作业指导书.docx.20260414_101530.bak
继续下一条。
```

#### 任务结束反馈
```text
本次任务结束。
共命中 8 条，已修改 5 条，跳过 2 条，失败 1 条。
已生成《体系文件批量修订报告》。 
```

## 8. 工具能力要求

### 8.1 Knowledge_Retrieval
用于跨文件检索目标语句，要求支持：
- 关键词检索；
- 相似句检索；
- 元数据返回；
- 多文件汇总。

### 8.2 Content_Locator
用于将命中项定位到文件内部具体位置，要求支持：
- 章节/标题层级提取；
- 页码/段落号定位；
- 表格单元格坐标定位；
- 上下文提取。

### 8.3 Diff_Viewer
用于生成修改前后对比，要求支持：
- Before / After 展示；
- 差异高亮；
- 可读性良好的预览卡片。

### 8.4 Confirmation_Controller
用于在写入前阻断流程并等待用户指令，要求支持：
- 单条确认；
- 跳过；
- 手动编辑；
- 中途终止。

### 8.5 File_Backup_Manager
用于在写入前自动生成备份，要求支持：
- 原文件复制；
- 备份命名规范；
- 备份路径可追溯。

### 8.6 File_Update_Writer
用于执行实际写入，要求支持：
- 定点替换；
- 替换前后校验；
- 错误回传；
- 失败不中断日志。

### 8.7 Revision_Log_Manager
用于记录全过程审计日志，要求支持：
- 检索日志；
- 确认日志；
- 修改日志；
- 回滚日志；
- 结果导出。

## 9. 风险控制

### 9.1 默认不直接整库批改
即使用户输入的是一句话，也必须先展示命中结果，禁止“检索到后立即自动全部覆盖”。

### 9.2 相似项单独标识
语义相近但不完全一致的句子，默认归类为“疑似项”，不能与精确命中项混写。

### 9.3 结构性内容谨慎处理
下列区域默认高风险：
- 页眉页脚
- 目录
- 版本记录页
- 修订历史表
- 模板占位区
- 受控编号区
- 表格公式区

高风险区域即使命中，也必须重点提示用户。

### 9.4 二进制文件与只读文件保护
- `.pdf` 默认不执行原位改写；
- 加密文件、锁定文件、签章文件默认只检索不写入；
- 对 `.docx`、`.xlsx` 需使用结构化解析和写回，不允许直接当纯文本粗暴替换。

### 9.5 回滚机制
一旦发现批量修订有误，应支持基于备份文件进行回滚，并写入回滚日志。

---

## 10. 异常处理

### 10.1 未检索到结果
输出：
- 未找到命中项；
- 建议用户检查原句是否存在空格、标点、换行或版本差异；
- 可建议改用模糊或语义检索。

### 10.2 检索到过多结果
若命中量过大，应提醒用户缩小范围，例如：
- 指定部门目录；
- 限制文件类型；
- 提供更长原句；
- 按批次处理。

### 10.3 文件写入失败
记录失败原因，例如：
- 文件占用；
- 权限不足；
- 文件格式不支持；
- 命中位置已变化；
- 替换后校验失败。

### 10.4 用户中途修改目标语句
如用户在流程中更换 `Target_Text` 或 `Replacement_Text`，则当前任务应结束，重新建立新任务，避免旧清单失效。

## 12. 推荐执行模式

### 模式 A：标准安全模式
适用于正式体系文件。
- 精确匹配优先；
- 一条一确认；
- 强制备份；
- 强制日志。

### 模式 B：审阅预演模式
适用于先看结果再决定。
- 全流程执行到预览；
- 不实际写入；
- 输出完整命中与 diff 清单。

### 模式 C：人工微调模式
适用于条款虽然相似，但不同文件措辞不能完全统一的情况。
- 每条都允许单独手动编辑新内容；
- 仍然保留统一检索与统一留痕。

---

## 13. 最小可用工作流（MVP）
1. 用户输入原句与新句；
2. 系统全库检索；
3. 输出命中文件清单；
4. 逐条展示命中位置与前后文；
5. 用户逐条确认；
6. 系统备份并写入；
7. 输出修订报告。

---

## 14. 成功判定标准
当且仅当满足以下条件时，任务才视为成功：
1. 已完成指定范围内的全局检索；
2. 每条修改前均经过用户确认；
3. 所有已确认条目都完成了备份与写入校验；
4. 已生成完整修订日志与结果报告；
5. 对失败项与跳过项均有明确记录。

---

## 15. 给 Dify / Coze / 自建 Agent 的落地说明
本 Skill 是“流程规则定义”，不是具体平台配置文件。

落地时应映射为以下能力模块：
- 检索节点：知识库 / 文件库检索
- 定位节点：段落、页码、单元格定位
- 审阅节点：生成 diff 卡片
- 交互节点：等待用户确认
- 写入节点：调用文件编辑 API
- 备份节点：创建备份副本
- 日志节点：写入审计记录
- 报告节点：生成修订报告

如果落地到 Dify 或 Coze：
- 检索环节建议用“关键词 + 向量”混合模式；
- 人工确认环节建议使用循环节点或对话式中断；
- 文件写入建议通过外部 API 完成；
- `.docx/.xlsx` 应由专门服务处理，不建议直接用文本替换。

---

## 16. 一句话总结
这个 Skill 的本质不是“批量替换”，而是“可审阅、可确认、可回滚、可审计的体系文件全局协同修订流程”。
